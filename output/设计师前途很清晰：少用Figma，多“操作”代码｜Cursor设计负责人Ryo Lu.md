## 1. Ryo 的設計觀：軟體不是像素，是一坨「概念」

- 軟體本質：
  - 不是一堆畫面、按鈕，而是：
    - 一組 **概念（concepts）**，和概念之間的 **關係（relations）**，
    - 這些概念在不同層級：資料結構、基礎設施（infra）、UI 渲染、使用者操作，一路向上貫通。
  - 理想情況下，如果你真的知道要做什麼、掌握所有資訊，整個系統「**理應存在一個更簡單、更本質的解**」。
- 很多 SaaS / 產品的共同點：
  - Asana = tasks & projects；Stripe Dashboard = 各種 object 的列表 + 詳情頁；TikTok = list of videos。  
  - 底層 pattern 幾乎都一樣：CRUD、列表、詳情、relationship、權限／組織層級等。
- 設計師的核心能力：
  - 不是只會畫 UI，而是：
    - **拆解出真正的概念單位**（units），
    - 理解它們的關係與最大可能性（maximum possibilities），
    - 不斷把系統往「更簡單、更統一、更可擴張」推進。

---

## 2. 從 skeuomorphism 到扁平化，再到「回到概念」

- 早期 GUI：
  - 把實體世界的隱喻（桌面、資料夾、相簿等）搬進電腦，
  - 擬真（Aqua、XP 等）是為了「**拉更多人進數位世界**」，讓他們能靠熟悉的物件理解抽象系統。
- 扁平化（flat design）：
  - 使用者已經熟悉概念後，就可以把多餘的擬真細節剃掉，
  - 回到「列表就是列表、grid 就是 grid」的更抽象形式，
  - **優點**：更接近概念本身、更簡潔易擴張；
  - **缺點**：少了很多 character／趣味，多餘卻好玩的東西消失。
- Ryo 的觀點：
  - 好的軟體需要在「**本質簡潔**」與「**有點 slack、可玩性**」之間找到平衡，
  - 不只功能上的 slack，也包含一些讓人想反覆玩耍的特質（動效、材質、節奏感）。

---

## 3. 為什麼 Figma 不是「真實」，程式碼才是材料？

- Figma 的限制：
  - 你在 Figma 做的只是 **靜態狀態** 和 **狀態切換**，
  - 看不到真實資料流、無法接上實際 API、看不到 runtime 的行為，
  - 卡在 `.fig` 檔與 WebGL canvas 抽象層上，**不是跑出來的東西**。
- 相對地，直接用程式碼（搭配 agent / IDE）：
  - 你可以：
    - 直接跟資料與邏輯互動（真實 input/output）、
    - 看系統在不同狀態下的表現、
    - 讓 agent 幫你做修改，再自己微調細節。
  - 工具越往「直接操作真實材料」走，越接近真實產品，也越能快速迭代。
- Ryo 實際做法：
  - 去年多數時候還是從 Figma 開始；最近一版 Cursor（他口中的第三版 baby Cursor）幾乎都在 **Electron App + Cursor CLI** 裡直接做原型：
    - 兩週內在程式碼中做出傳統流程要 2–3 個月才做得出的 prototype；
    - 可以一邊寫、一邊試、一邊調整邏輯與 UI 細節。

## 4. Builder 取代傳統「設計師 + PM + 前端」分工

- 過去的分工：
  - 設計師在 Figma 搓像素 → 寫 PRD / spec → 給 PM、EM 排優先順序 → 工程實作 → 之後很少再被打磨。
  - 問題：
    - 設計永遠停在「一次性交付」而非持續演化；
    - 很多 edge case / 系統性問題在 Figma 裡根本看不到。
- 未來（其實已經在發生）：
  - **角色融合成 Builder**：
    - 可以看大局（系統、roadmap），也能直接下手改一個歪掉的字級或 grid；
    - 直接在 codebase 上動手，和 agent 一起完成 50–70% 的工作，剩下的是人的判斷和品味。
  - 工具側（例如 Cursor）會：
    - 把 Git / branch / npm / localhost 這些概念包起來，
    - 讓非工程背景的人可以「先做出東西，再慢慢理解底層概念」。
- 本質：
  - 過去職能區隔（Designer / PM / Engineer）很多時候是 **組織妥協**，不是本質，
  - 在 AI + agent 工具成熟後，大家其實都在「寫軟體」，只是重心不同而已。

## 5. 對設計師的具體建議：從「搓像素」轉向「進化概念」

### 5.1 mindset / 習慣 / skill 的順序

- 先有 **mindset（想用 code 直接做實物）**，
- 再養成 **habit（常態用 Cursor / agent 來做 prototype）**，
- skill 自然會在反覆 build 中長起來（包括 debug、API 調用、deploy 等）。

### 5.2 具體行動建議

- **開始用 Cursor／agent 做 side project**：
  - 不要期待第一個 prompt 就完美，
  - 學會：
    - 把模糊需求拆成清楚的 context / component / criteria 給 agent，
    - 用 plan mode 展開，review、加細節，
    - 用 design system 約束 agent，不要每次重新造輪子。
- **接受「不舒服期」是必經階段**：
  - 從 Figma 複製 artboard 的快感，變成和 agent 來回對話、等回應、用 version control 管多個方案，
  - 這中間的 latency／不適應是不可避免的，但只要過了門檻，設計空間大幅擴張。

### 5.3 审美與系統感：仍然是設計師的核心 moat

- Ryo 對審美的看法：
  - 美感不是純主觀，而是對「**系統邏輯與韻律**」的深度體感：
    - grid、type scale、spacing、色彩使用、對齊方式、斷行規則等等，
    - 使用者不一定講得出來，但能感覺到「舒服、不塑膠、不 AI slop」。
  - 好的設計師會：
    - 看到那些「本來應該一樣卻不一樣」的地方並想統一，
    - 看到那些「不必要卻很重」的東西並想刪掉或重構。
- 如何養這種能力：
  - 用「**活生生的系統**」視角看軟體，就像看自然生物系統：
    - 每個 layer 有自己的簡單規律，上下層又是同一個東西的不同表現；
  - 去看歷史上的好設計（家具、版式、建築、軟體），
  - 在各種領域（音樂、藝術、生物學）裡找共通 pattern，訓練自己壓縮與抽象的習慣。

## 6. Ryo 招設計師的標準（你要成為什麼樣的人）

- **最低門檻**：
  - 視覺層完全沒問題：
    - 字級、行距、版式、節奏、對齊、顏色、圖片裁切等，做出來就是順、乾淨、有韻律感。
- **加分向量（不必全滿，但要有自己的方向）**：
  - 很會處理複雜 flow、edge cases，讓使用路徑順到底；
  - 對設計系統（design system）底層規則很敏感；
  - 對技術（程式碼、infra、甚至訓模型）有高度興趣與實作經驗；
  - 願意直接在 codebase 裡構建，而不是只在 Figma 裡畫。
- 組合觀：
  - 他希望團隊成員 **背景／國家／直覺都不一樣**，
  - 但都能把整個系統裝進腦袋裡，對同一個「軟體本質」對齊方向。

---

## 7. 給華語設計師的一句話

- **「開始用程式碼做東西。」**
  - 不要期待第一次就對，不要怕 agent 出錯，
  - 讓自己從「交付設計稿」變成「持續進化系統的人」，
  - 從此你做的就不再是畫面，而是活的系統與概念本身。
